1. What does a DOCTYPE do?

Short form of 'Document Type,' it specifies which Document Type Definition (DTD) should be followed.

For the case of web-pages, you can specify which HTML version the document follows. To specify that an html file respects HTML5 standards, we declare <!DOCTYPE html>. If the DOCTYPE is not specified, then quirks mode will be triggered. In quirks mode, content may display in unexpected ways. If it is specified, then no-quirks mode is activated.

2. How do you serve a page with content in multiple languages?

When an HTTP request is sent to a server, the request usually contains info about language preferences (Accept-Language header). The server uses this information to return a version of the HTML document to match the language needed. Usually, the html will have an attribute lang. E.g. <html lang="en"></html>

3. What kind of things must you be wary of when designing/developing for multilingual sites?

Some things that should be kept in mind:

  1. Using lang attribute in HTML, to specify language.

  2. UX: Allow users to easily change into their native language, possibly by modal.

  3. Text in images are NOT scalable.

  4. Sentence length: some sentences are much longer in some languages than others.

  5. Date and currency formatting are different for some countries/langues.

  6. Text reading direction: some are right-to-left, and others are left-to-right.

4. What are data- attributes good for?

A method of storing extra data before JavaScript frameworks became more mainstream. Using data- attributes allowed to store data in the DOM itself. It is not recommended because users can simply change these attributes via inspect element.

5. Consider HTML5 as an open web platform. What are the building blocks of HTML5?

New and important stuff in HTML5:

  1. Semantics: more specific ways to define content. New tags such as <footer> and <aside> to better describe content.

  2. Connectivity: new ways to communicate with server. Web sockets, server-sent events (as compared to client having to send data first).

  3. Offline and Storage: webpages can now store client-side data more efficiently. Multimedia: video and audio now have their own tags. Thus can operate offline better.

  4. 2D-3D graphics/effects: more diverse presentation. Canvas, SVG, WebGL.

  5. Performance and Integration: for better speed and optimization. Web workers for background threads (!). History API, manipulating browser history.

  6. Device Access: More options for input and output. Styling: extension of CSS to CSS3. Touch events, camera API, device orientation.

6. Describe the difference between a cookie, sessionStorage, and localStorage.

All three are key-value storage options on the client side. They can store values as strings (must use JSON.stringify and JSON.parse for objects).

cookie: Client or server can initiate storage. Servers must use 'Set-Cookie' header. Data expiry is set by initiator, and can be set to persist across browser sessions (or otherwise). Cookies are sent to server with every HTTP request, with a 'Cookie' header. They can hold 4kb per domain, and are accessible by any window.

localStorage and sessionStorage similarities: Initiated on the client side. Data stored in either are not sent to server with every HTTP request. They both store 5 MB per domain.

localStorage: Once set, lasts forever (until removed by user or some application), thus it is persistent across browser sessions. It is accessible by any window.

sessionStorage: Once set, lasts until tab closes. Data doesn't persist across browser sessions, and is only accessible by the same tab in a window.

7. Describe the difference between <script>, <script async> and <script defer>

<script> : HTML parsing is blocked and the script is downloaded. The script must finish downloading before parsing resumes.

<script async> : HTML keeps parsing, and script is downloaded in parallel. The script is executed once it is available. It is a good idea to use this when the script is independent of other scripts, or the DOM.

<script defer> : HTML keeps parsing, and script is downloaded in parallel. The script is executed once HTML is done parsing. It is a good idea to use this when script relies on DOM.

8. Why is it good to position CSS <link> between <head></head > tags, and JS <script > just before </body> ? Any exceptions?

CSS : When placing <link> inside the <head></head> tags, then the HTML and CSS are being parsed simultaneously. Both the DOM and CSSOM are created, and used to create the website visuals. It also allows for progressive rendering. If the CSS link is placed at the bottom, the styles are parsed after the HTML is... which will prevent progressive rendering in many browsers. This is because browsers block rendering so as to not have to restyle elements as their styles change.

SCRIPT : Using just <script> halts HTML parsing until the script has been downloaded. This means users have to wait (not good), so the solution is to load scripts after HTML is parsed, before the ending </body> tag. Using <script defer> in the head is better and achieves the same effect, but there are a few browsers that don't support it.

9. What is progressive rendering?

Progressive rendering is are the sets of techniques to render content for display as fast as possible. This is to improve webpage performance.

Some examples of progressive rendering:

  1. Image Lazy Loading: only loading images once they are about to be displayed.

  2. Above-the-fold Rendering: prioritize rendering content that is going to be visible immediately. Other content can be rendered after, or later.

  3. Async HTML fragments: Rendering HTML elements to the browser, while the page is made in the back-end.

10. Why would you use a srcset attribute in an image tag? Explain how the browser evaluates the content of this attribute.

Using srcset is for when we want to use differently sizes images depending on the user's device width. We display higher resolution images for higher resolution displays, and otherwise.

Use it like so:
<img srcset="small.png 500w, medium.png 1000w, large.png 2000w" src="...">

Where the number beside 'w' denotes the width of that image in pixels. Then for device width of 400px, srcset makes these calculations:
500/400 = 1.25
1000/400 = 2.5
2000/400 = 5

And it chooses the image for which the evaluated number is closest to 1. That would be small.png with 1.25 for this case.
